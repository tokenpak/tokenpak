# `tokenpak/_snapshots/`

This directory holds **deterministic snapshots** of repository state that the
release-gate trust contract pins for ratchet enforcement. Each snapshot is a
machine-readable file checked into git. When the underlying source changes,
the corresponding snapshot must be regenerated and committed; CI / pre-merge
review then verifies that the snapshot is up to date.

The ratchet protects against silent removal of workflow steps, public API
symbols, and other surface elements that must remain stable across releases.

## Files in this directory

### `workflow-steps.json` (P5b)

Sorted snapshot of every step across every workflow under `.github/workflows/`.

Schema (one row per workflow step):

```json
{
  "workflow": "release.yml",
  "job": "build",
  "step_idx": 0,
  "step_id": null,
  "step_name": "actions/checkout@v4"
}
```

Fields:

- `workflow` — basename of the workflow file (e.g. `release.yml`).
- `job` — the job key under `jobs:` (e.g. `build`, `publish`).
- `step_idx` — zero-based position of the step within its job (preserves order).
- `step_id` — the step's `id:` field if set, else `null`.
- `step_name` — the step's `name:` field; if absent, falls back to `uses:`; if
  neither, `<unnamed>`.

Sort order: `(workflow, job, step_idx)`. JSON-serialized with `indent=2`,
`ensure_ascii=False`, trailing newline. Regenerated by
[`scripts/snapshot-workflow-steps.py`](../../scripts/snapshot-workflow-steps.py).

## Ratchet protocol

The ratchet is enforced via two Makefile targets and (where wired) a CI check.

### Local commands

```bash
# Regenerate the snapshot from the current workflow set.
# Use this BEFORE opening a PR if your PR touches any .github/workflows/*.yml file.
make workflow-steps-snapshot

# Check that the committed snapshot matches the current workflows.
# Exit 0 = no drift; exit 1 = drift detected.
make workflow-steps-check
```

### When the ratchet flags drift

If `make workflow-steps-check` fails on your branch, one of two things is
true:

1. **You added or renamed workflow steps.** Run `make workflow-steps-snapshot`
   and commit the regenerated JSON file. The ratchet will then pass.

2. **You removed workflow steps.** You must additionally declare each removed
   step in your PR body using the `removes-ci-step:` trailer (one line per
   removed step). Example PR body trailer block:

   ```text
   removes-ci-step: <step.id-of-removed-step>
   removes-ci-step: <step.id-of-another-removed-step>
   ```

   Then run `make workflow-steps-snapshot` and commit. The PR review verifies
   that every step missing from the new snapshot has a matching
   `removes-ci-step:` declaration in the PR body.

This protects against the failure mode where a refactor silently drops a
release-protecting step (dist-purity guard, identity-language check, etc.)
without the reviewer noticing.

### Allowed-companion classification

Per the workflow-as-code refactor isolation rule:

- The snapshot file under `tokenpak/_snapshots/` is an allowed-companion
  surface for workflow PRs — a PR that touches `.github/workflows/*.yml` MAY
  also touch this snapshot in the same PR.
- The generator script under `scripts/snapshot-workflow-steps.py` is also
  allowed-companion.
- Workflow PRs MUST NOT touch source code outside the workflow files,
  `pyproject.toml`, `Makefile`, `.changeset/*`, this snapshot directory, and
  `release-notes-template.md`.

## Future snapshots

Other ratchet-protected snapshots may land in this directory under separate
PRs:

- `public-api.json` — pinned public-API surface; ratchet enforces declared
  symbol removals. Referenced by the release-gate trust contract initiative-
  start gate.
- `dist-purity-rules.json` — pinned dist-purity assertion rules; ratchet
  enforces that `dist/` checks remain in sync between the `build` and
  `publish` jobs.

These are out of scope for the current PR; they will land in their own
release-gate hardening initiatives as the trust contract continues to mature.
