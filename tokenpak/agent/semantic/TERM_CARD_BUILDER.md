# TokenPak Term-Card Builder — Developer Guide

## Overview

The **Term-Card Builder** is the semantic layer for the TokenPak knowledge base. It creates, validates, and maintains compact **5W1H micro-cards** for deterministic term resolution — a thin vocabulary index sitting above the primary auto-indexed KB blocks.

Location: `tokenpak/agent/semantic/term_card_builder.py`  
Card store: `tokenpak/term_cards.json`

---

## Card Schema

Each card is stored under its canonical snake_case key in `term_cards.json`:

| Field | Type | Cap | Required |
|-------|------|-----|----------|
| `term` | str | 60 chars | ✓ |
| `what` | str | 120 chars | ✓ |
| `who` | str | 80 chars | ✓ |
| `where` | list[str] | 5 items × 60 chars | ✓ |
| `why` | str | 120 chars | ✓ |
| `how` | str | 150 chars | ✓ |
| `not_this` | str | 120 chars | optional |
| `aliases` | list[str] | 6 items × 40 chars | ✓ |
| `tier` | int | 0, 1, or 2 | ✓ |
| `confidence` | float | 0.0–1.0 | ✓ |
| `source_refs` | list[str] | 10 items × 80 chars | ✓ |
| `updated_at` | str | ISO-8601 | recommended |

Field caps are **hard limits** — the builder truncates to cap-1 and appends `…` if a value exceeds the limit.

---

## Tiering Policy

| Tier | Name | Volume | Source | Confidence |
|------|------|--------|--------|------------|
| **0** | Hand-curated | 100–300 target | Human-authored | 1.0 |
| **1** | Auto-generated | Unlimited | Source scan | 0.40–0.75 |
| **2** | Lazy on-demand | As needed | Runtime log hit | 0.1 (stub) |

- **Tier 0** cards are never overwritten by the builder — human authorship takes priority.
- **Tier 1** cards are generated by scanning Python source for high-frequency identifiers (≥3 occurrences). They are stubs that should be reviewed and promoted to Tier 0 or deleted.
- **Tier 2** cards are created on-demand when an unknown term is queried. They are stubs pending enrichment.

## CLI Usage

All commands are run from the repo root (`~/tokenpak/`):

```bash
# Full Tier 0/1 build (normalise + scan + save)
python3 tokenpak/agent/semantic/term_card_builder.py build

# Build without Tier 1 auto-extraction (Tier 0 only)
python3 tokenpak/agent/semantic/term_card_builder.py build --no-tier1

# Dry-run (no disk writes, full report)
python3 tokenpak/agent/semantic/term_card_builder.py build --dry-run

# Validate existing cards (exit code 1 if errors or conflicts)
python3 tokenpak/agent/semantic/term_card_builder.py validate

# Print validation report
python3 tokenpak/agent/semantic/term_card_builder.py report

# Lazy-add a Tier 2 stub
python3 tokenpak/agent/semantic/term_card_builder.py lazy-add "my_term"
python3 tokenpak/agent/semantic/term_card_builder.py lazy-add "my_term" --source-refs foo.py bar.py
```

## Regeneration Workflow

Use this checklist when regenerating or reviewing cards:

### 1. Sync vault + pull latest source
```bash
cd ~/vault && bash scripts/vault-sync.sh
cd ~/tokenpak && git pull
```

### 2. Dry-run build to preview changes
```bash
python3 tokenpak/agent/semantic/term_card_builder.py build --dry-run
```
Review the report — check for alias conflicts and unexpected Tier 1 additions.

### 3. Build and save
```bash
python3 tokenpak/agent/semantic/term_card_builder.py build
```

### 4. Review new Tier 1 cards
Open `tokenpak/term_cards.json` and search for `"tier": 1` entries. For each:
- Promote to Tier 0 with full 5W1H fields if it's a real domain term
- Delete if it's noise (internal variable, stdlib name, etc.)

### 5. Validate
```bash
python3 tokenpak/agent/semantic/term_card_builder.py validate
```
Fix any reported alias conflicts before committing.

### 6. Commit
```bash
git add tokenpak/term_cards.json
git commit -m "kb: regenerate term cards — $(date +%Y-%m-%d)"
git push
```

## Alias Conflict Resolution

The builder **always** reports alias conflicts — it will never silently skip them.

A conflict occurs when the same alias string (case-insensitive) is claimed by two or more canonical terms. To fix:

1. Run `validate` to get the conflict list
2. Decide which canonical term "owns" the alias
3. Remove the alias from the losers' `aliases[]` lists
4. Re-run validate to confirm clean

## Python API

```python
from tokenpak.agent.semantic import (
    build, lazy_add, load_cards, validate_card,
    detect_alias_conflicts, validation_report
)

# Load cards
cards = load_cards()

# Run full build
cards = build(source_root=Path("~/tokenpak/tokenpak"))

# Lazy-add a term at runtime
stub = lazy_add("unknown_term", cards, source_refs=["proxy.py"])

# Validate a single card
errors = validate_card(cards["baseline_cost"])  # [] = valid

# Detect alias conflicts
conflicts = detect_alias_conflicts(cards)  # [] = clean

# Full report string
print(validation_report(cards))
```

## Output Determinism

Cards are stored with **deterministic ordering**: sorted by `(tier ASC, term ASC)`. This means:
- Git diffs are clean and reviewable
- SHA256 of the file is stable for the same logical content
- CI can verify no unintended changes with `git diff --exit-code`
