## Lock Acquisition Order Analysis

### Key Finding: **No Multi-Lock Functions**

Scan of all functions shows **no function ever holds more than one lock simultaneously**. This eliminates the primary source of deadlock risk.

- Each lock guards a single resource
- Locks are acquired and released within the same function
- No nested lock patterns detected

### Lock Dependency Graph

Since no function holds multiple locks, there are **no circular dependencies** and **no ordering constraints to violate**.

```
✓ Lock A → resource A (independent)
✓ Lock B → resource B (independent)
✓ ...
✓ Lock N → resource N (independent)
```

**Verdict:** Low deadlock risk. Ordering violations = 0.

## Async DB Writes Architecture ✅

The proxy has already implemented async DB writes, removing the synchronous blocking from the request path:

### Current Implementation

1. **Request path (non-blocking):**
   ```python
   def Monitor.log(...):
       _start_db_writer()  # Ensure writer thread active
       _DB_WRITE_QUEUE.put((db_path, insert_data))  # Queue the write
       return  # ← returns immediately, does NOT wait for DB
   ```

2. **Writer thread (background):**
   ```python
   def _db_writer_thread():
       while not _DB_WRITER_STOP.is_set():
           task = _DB_WRITE_QUEUE.get(timeout=1)
           db_path, insert_data = task
           conn = _get_db_connection(db_path)
           conn.execute(INSERT ...)  # SQLite INSERT happens here
           conn.commit()
   ```

3. **Thread lifecycle:**
   - Spawned on first `log()` call via `_start_db_writer()`
   - Runs as daemon thread
   - Gracefully shuts down via `_DB_WRITER_STOP` event

### Impact on Request Latency

- **Before:** Request blocked on SQLite INSERT
- **After:** Request completes in ~1μs (queue.put), DB write in background

**Estimated latency improvement:** 10–100ms per request (depends on DB I/O)

## Dead Code: `_DB_LOCK`

### Issue
```python
_DB_LOCK = threading.Lock()  # Line 1969
# Never used anywhere else in codebase
```

### Why It Exists
Legacy artifact from earlier synchronous DB implementation. When DB writes were blocking the request path, `_DB_LOCK` would have guarded the write operation. Now that writes are fully async via queue, the lock is unnecessary.

### Action Items
- [ ] **Verify:** `_DB_LOCK` is truly unused (no conditional imports, no debug code)
- [ ] **Remove:** Delete line 1969
- [ ] **Test:** Run full test suite to confirm no regressions
- [ ] **Commit:** `"trix: remove dead _DB_LOCK (superseded by async queue)"`
