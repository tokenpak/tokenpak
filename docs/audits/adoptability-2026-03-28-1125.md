```

```

**Core accuracy concern — StageTrace bug:**
Every request triggers a pre-pipeline fallback. This means the compression pipeline's StageTrace instrumentation is failing, and the proxy falls back to the original body. The question is: which pipeline stages depend on StageTrace and are therefore being skipped?

If StageTrace is only instrumentation (tracing/telemetry) and the actual compression steps run independently of it, accuracy is fine. If pipeline stages gate on StageTrace metadata, those stages are being silently bypassed. From context, "falling back to original body" suggests the entire compression pipeline is being skipped for requests that trigger this error — meaning these requests get NO compression, just pass-through.

**Canon inconsistency across fleet:**
- Sue: canon enabled
- Trix: canon disabled  
- Cali: canon disabled

This is a fleet config drift. Canon deduplication is a meaningful compression feature — having it disabled on 2/3 agents is either intentional (and should be documented) or a config regression.

**Validation gate:**
Repeated soft-block: "deterministic request missing required context block". This is informational — requests still forward. But it fires consistently, suggesting a class of requests that could be better optimized.

**Tree-of-Thought Analysis:**

*Angle 1 — Is user output getting corrupted?*
With 83.2% protected tokens and 1.0 stability score, the proxy is passing most content through unmodified. The protection-first approach ensures correctness at the cost of compression efficiency. Users are unlikely to see degraded output quality.

*Angle 2 — What's the StageTrace blast radius?*
If the fallback to "original body" means zero compression for affected requests, then the 3.69% compression ratio is achieved only on the requests where StageTrace doesn't error. Looking at the by-model data: opus achieves 10.82% compression, sonnet 6.61%. These are the models that get more complex, compressible requests. This suggests StageTrace may only fire on certain request patterns, not all requests — or the compression happening is in a stage that runs before StageTrace initializes.

*Angle 3 — Canon disabled impact:*
Canon catches repeated content across turns (e.g., system prompts that appear in every message). With canon disabled, those repeated tokens get sent every time. For a busy session with 50+ turns, this is meaningful overhead. Sue has it enabled and processes 7,078 requests at higher volume — this may explain why Sue's per-request stats look efficient.

## New Issues Found

### 🔴 HIGH — StageTrace() bug fires on every request (persistent, not new but worse)
**Finding:** `StageTrace() takes no arguments` appears in logs on essentially every request processed by the proxy over the past 4+ hours. This is a code error where a class is instantiated with arguments that its `__init__` doesn't accept. The proxy falls back gracefully, but the compression pipeline is bypassed for affected requests.
**Impact:** Unknown fraction of requests receive zero compression. The 3.69% compression ratio is being measured on pass-through requests OR on requests that don't trigger this error path.
**File:** Somewhere in the compression pipeline that initializes `StageTrace(...)` — search `proxy.py` for `StageTrace(`.

### 🟡 MEDIUM — Cali p99 = 138,979ms
**Finding:** Cali's `/health` endpoint reports p99 latency of 138,979ms (2.3 minutes) from 100 latency samples.
**Impact:** Roughly 1% of Cali's 3,638 requests today took 2+ minutes. With Cali processing horse racing pipeline and picks generation, this could be causing user-visible delays.
**Recommended investigation:** Check Cali journal for requests around the p99 window; check if vault reload is causing lock contention on large requests.

### 🟡 MEDIUM — Canon config drift across fleet
**Finding:** Sue has canon enabled; Trix and Cali have it disabled. Canon deduplication is a meaningful compression feature.
**Impact:** 2/3 agents are missing cross-turn deduplication savings. Either this is intentional (and should be documented in fleet config) or it's a regression.

### 🟡 MEDIUM — 331 flaky tests in full parallel suite
**Finding:** Running the full test suite with xdist produces 331 failures. Running affected tests (adoption_killers, vault_reload, vault_search) in isolation: all pass.
**Impact:** CI alarm fatigue. Hard to distinguish real regressions from noise. The BenchmarkWarning about xdist disabling benchmarks is also suppressing performance regression detection.

### ℹ️ LOW — BM25 vault search timings not surfaced in /stats
**Finding:** `vault_last_timing_ms: {}` in /stats session data — vault retrieval timings aren't being logged to the stats endpoint. Can't confirm whether BM25 search is hitting the <50ms internal target.

### ℹ️ LOW — Key pool shows 0 Anthropic keys at startup
**Finding:** `[key-pool] Found 0 Anthropic API key(s)` on every startup. The proxy operates because OpenClaw injects keys via its config. But for standalone TokenPak users, this silent "0 keys" message would be confusing and non-actionable.
