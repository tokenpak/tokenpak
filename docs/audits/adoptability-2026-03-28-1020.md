```
10:18:20 → Failed: timeout       (restart #9)
10:18:59 → Failed: protocol      (restart #10)
10:19:04 → Restart counter: 10  ← still activating at audit time
```

**Failure pattern analysis:**

*`timeout` failures* — WatchdogSec=30 in the service unit. The proxy uses `Type=notify`, meaning it must call `sd_notify(READY=1)` within 30 seconds of start. If startup takes longer (e.g., due to vault index loading 7,938 blocks at 33M tokens), systemd kills it.

*`protocol` failures* — sd_notify sent before the socket was ready, or with malformed payload. This alternating pattern suggests the process sometimes starts fast enough to call notify but with incorrect state.

**Root cause hypothesis:** The pytest test suite spawned during this audit (~10:07 AM) created CPU/memory pressure. Vault index loading (33M tokens, 7,938 blocks) under load is pushing startup past the 30s watchdog deadline.

**Current status at audit time:**
```
Active: activating (start) since 10:18:59 PDT; 51s ago
Memory: 708.2M (swap max: 1.0G, swap: 30.2M)
Restart counter: 10
```
After restart counter = 10 within StartLimitBurst=5 / StartLimitIntervalSec=300, systemd may stop retrying — this is the critical risk.

**Fleet health (SSH checked):**
| Agent | Status | Requests Today |
|-------|--------|---------------|
| Trix | 🟡 Restarting | — |
| Sue | ✅ ok | 4,090 |
| Cali | ✅ ok | 3,439 |

*Note: Sue and Cali health endpoints don't surface p50_latency_ms — this is a health endpoint inconsistency.*

**Positive signal:** 0 application-level errors in the logs. This is an infrastructure/startup issue, not a correctness issue.

**Tree-of-Thought Analysis:**

*Angle 1 — Is this transient or systemic?*
10 restarts in 2 hours suggests a repeated trigger. The pytest run is the most likely culprit — it creates massive I/O and memory pressure on a machine that already runs tight (708MB RSS + swap). The proxy likely starts fine when not under load.

*Angle 2 — What's the user impact?*
Each restart takes 37–95 seconds. During that window, all requests to the proxy fail. With 4,502 daily requests, even 10 minutes of downtime is significant. The `Restart=always` ensures recovery, but the cycle time is long.

*Angle 3 — What's the fix?*
Two options: (a) Increase `WatchdogSec` from 30s to 120s to accommodate slow startup under load, or (b) disable `Type=notify` temporarily and use `Type=simple` so systemd doesn't wait for sd_notify. Option (a) is safer.
