tool name: read_active_file

tool name: read_tagged_files
tool description: Returns the contents of all files the user has tagged into the conversation via `@path/to/file`. Each tagged file comes with its full path, content, and detected language. Tagged files persist across turns until the user untags them. Use this to ground multi-file refactors. Returns a map from path to file metadata + content.
tool parameters: {}

tool name: search_workspace
tool description: Search the user's workspace using a ripgrep-compatible regex. Returns matching files with line numbers and surrounding context. Supports glob filtering and language-type filtering. Prefer specific patterns; broad searches return noise that the user pays for. Maximum 200 results — use a tighter pattern if you hit the cap.
tool parameters: {pattern: string, glob: string optional, type: string optional, max_results: integer optional}

tool name: open_file
tool description: Open a file in the editor at a specific line range. Does not return content — read_active_file or read_tagged_files for content access. Useful when you want to direct the user's attention to a specific location in the codebase before proposing an edit. Honors the user's preference for split panes.
tool parameters: {path: string, line: integer optional, end_line: integer optional}

tool name: propose_edit
tool description: Propose a code edit to the user. The user reviews the diff and either accepts or rejects it. The edit is specified as a path, an old_string that must appear exactly once in the file, and a new_string that replaces it. For multi-file edits, propose each edit separately — the user can accept each independently. Never use this tool for changes outside the user's project root or for changes the user did not request. Return the edit's status (accepted / rejected / pending) when the user has decided.
tool parameters: {path: string, old_string: string, new_string: string, rationale: string optional}

tool name: run_terminal_command
tool description: Run a shell command in the user's project root. The user's shell environment and PATH are inherited. Commands run with a 30-second timeout by default. Returns stdout, stderr, and exit code. Long-running commands (dev servers, watch modes) should be invoked with `--timeout 0` to detach; their output is monitored asynchronously and surfaced when the user asks. Never run destructive commands (rm, force-push, drop) without explicit user authorization.
tool parameters: {command: string, timeout_seconds: integer optional, working_directory: string optional}
